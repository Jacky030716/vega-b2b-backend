# AI Governance

The backend records governed AI activity through `ai_audit_logs`.

## Use Cases

- `CUSTOM_CHALLENGE_EXTRACTION`
- `MODULE_CHALLENGE_PLANNING`
- `SPELL_CATCHER_CONFIG`
- `SYLLABLE_SUSHI_CONFIG`
- `VOICE_BRIDGE_CONFIG`
- `ADMIN_AUDITOR`
- `STICKER_GENERATION`

## Prompt Registry

Prompt definitions are code-based in `AiPromptRegistry`. Each prompt has a use case,
version, description, system instruction, and output schema name. Prompt wording is
centralized there so future Malaysian syllabus-aligned variants can be versioned
without changing endpoint contracts.

## Audit Lifecycle

Each governed call starts with a `PENDING` audit row. The caller then records:

- `VALID` when output passes validation
- `INVALID` when the provider responded but schema validation failed
- `FAILED` when the provider or downstream generation failed

Input, raw output, parsed output, validation errors, and related user/classroom/module
or challenge IDs are stored as JSON where applicable.

The audit table stores:

- `id`
- `use_case`
- `provider`
- `model_name`
- `prompt_version`
- `input_payload_json`
- `raw_output_json`
- `parsed_output_json`
- `validation_status`
- `validation_errors_json`
- `related_user_id`
- `related_classroom_id`
- `related_module_id`
- `related_challenge_id`
- `created_at`

Current providers are `GEMINI` for text JSON generation, `HUGGING_FACE` for sticker
image generation, and `RULE_BASED` for the deterministic adaptive game config stage.

## Validation

AI output is validated before it is used:

- custom challenge drafts are validated against game-specific draft shapes
- module plans reject invalid game types, invalid difficulty, empty selections, and
  selected words outside the module vocabulary
- admin auditor responses must include an answer and numeric `matchedUserIds`
- sticker generation audits the image provider response and downstream storage result
- module game config generation is currently rule-based and logged as such

## Challenge Provenance

Challenges keep `sourceType` for curriculum origin, and add AI provenance fields:

- `isAiGenerated`
- `aiGenerationStatus`: `NONE`, `AI_ASSISTED`, `AI_GENERATED`, `FAILED_FALLBACK`
- `aiUseCase`
- `aiAuditLogId`

Predefined module challenges generated with Gemini planning keep
`sourceType = PREDEFINED_MODULE` and set `aiGenerationStatus = AI_ASSISTED`.
Custom AI challenge drafts return `aiAuditLogId`; when the save request includes it,
the final challenge stores `aiUseCase = CUSTOM_CHALLENGE_EXTRACTION` and links to
that audit row.

## Module Generation Fallback

Module challenge generation calls Gemini only for word selection, difficulty, and
reasoning. The final playable config is generated by the existing rule-based adaptive
builder. If Gemini fails or returns invalid output, the module challenge falls back to
the existing rule-based generator and records the failed AI audit.

## Current Boundary

This layer does not add new AI gameplay. Malaysian syllabus-aligned intelligence will
build on these audit, registry, validation, and provenance foundations in a later pass.
